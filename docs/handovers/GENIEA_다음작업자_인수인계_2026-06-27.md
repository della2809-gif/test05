## 6. 작업자가 절대 놓치면 안 되는 표현/관점

1. “판단/평가”보다 “선택지를 넓혀주는 메이트”
   - 지니아 정체성은 압박/푸시가 아니라 선택지를 넓혀주는 메이트

2. 사용자님 말투는 부드럽게 돌려 말하는 상담사 톤이 아님
   - 직설적
   - 빠름
   - 바로 핵심을 묻고 전환함
   - 하지만 공격이 아니라 의미 전환과 선택지 확장

3. 소비자/B 역할
   - 사업자가 아니어도 작은 중간 역할을 줄 수 있음
   - 소비자가 말하는 반응은 관계 맥락에 따라 다름
   - “소비자가 거짓말한다”가 아니라 “관계에 따라 솔직하게 말하는 정도가 다르다”

4. 제품 상담
   - 병원/의료 진단은 분리
   - 영양·항산화·보조관리 관점은 설명
   - 약효 단정 금지
   - 내부용이라도 최소 안전문구 필요

5. DB 작업
   - 원본 그대로 넣지 말 것
   - DB별로 분리
   - 검색 가능한 질문 예시/키워드/상황을 넣을 것
   - 실제 반영 전 검토표/입력안 → 승인 → 반영

6. 레퍼런스 DB
   - 원본 허브 역할
   - 모든 원본이 곧바로 유저 답변에 노출되면 안 됨
   - 활성/비활성, 카테고리, 검토상태, 사용범위가 필요

## 7. 발화자 역할 반영 후 재검토 보강

상세 재검토 문서:
`/Users/kimnayeon/Desktop/chatgpt/docs/GENIEA_참여자역할_반영후_재검토_2026-06-27.md`

정정: 여기서 말하는 `참여자 역할`은 GENIEA 서비스 전체의 사용자/권한 설계가 아니라, Slack 스레드·전사본·현장 메모에서 `누가 어떤 입장으로 말하는지` 구분하기 위한 `발화자 역할 라벨`이다.

따라서 모든 사람 이름을 일일이 구분하거나 권한표로 만들 필요는 없다. 이름은 특정 사례 판단에 필요할 때만 남긴다.

### 7-1. 구분해야 하는 것은 사람 목록이 아니라 발화자 역할임

주요 발화자 역할:
- 김나연님/대표님/요청자: 지시, 정정, 운영 기준, DB 반영 기준을 주는 발화
- GENIEA/GD/봇: 답변, 정리, 검토표 생성. 잘못 이해한 답변과 정정된 기준을 구분해야 함
- 매니저님: 실제 DB 반영, 대량 수정, 배포 전 승인 맥락에서만 중요
- 현장 리더/자료 제공자: 예를 들어 수정 리더님, 언주님 등. 자료 제공이나 현장 코칭 맥락에 영향 있을 때만 기록
- 파트너: 같이 사업할 사람. 소비자 후속관리와 다르게 사업 가이드/교육 연결/불안 해소가 필요
- 소비자: 제품 섭취/건강관리 대상. Product/FAQ/Link/건강상담 말버전과 연결
- 소개자/B: 소비자와 상담자 사이에서 반응을 전달하는 중간 역할. 사업자가 아니어도 작은 B 역할 가능
- 스폰서/A 또는 상위 리더: 라인십, B-C 분리/보호, 스폰서십 코칭 맥락
- C/다운라인/상담 대상: ABC 구조에서 실제 케어 받거나 사업을 이어갈 사람
- 동석자/화자 불명: 전사본에서 누군지 불명확한 경우. 억지로 이름 확정하지 말 것

### 7-2. 이름을 기록해야 하는 경우

모든 사람 이름을 다 구분하지 않는다. 아래 경우에만 이름/상태를 남긴다.

1. 특정 사례의 상태가 다음 행동 판단에 중요할 때
   - 예: 박정민/구유빈/김현지 파트너 상태별 가이드

2. 자료 제공 역할이 중요할 때
   - 예: 언주님은 Image DB 업로드/현장 분류 방식 제공자

3. 스레드 역할 구분에 필요할 때
   - 예: 김수정 스레드, 이언주 스레드, 파트너+지니아 미팅 스레드

4. 승인/책임 라인에 필요할 때
   - 예: 매니저님 `<@U9E0YD4JC>`은 실제 DB 반영 승인자

### 7-3. 누락 여부는 `사람 이름`이 아니라 `역할 맥락` 기준으로 본다

확인 기준:
- 이 말은 누가 한 말인가?
- 실제 사용자 지시인가, 봇의 해석인가?
- 소비자 발화인가, 파트너 발화인가, 스폰서/리더 발화인가?
- 전사본 속 화자가 확실한가, 추정인가?
- 역할이 바뀌면 답변/DB 후보도 바뀌는가?

예:
- `파트너가 생겼다`는 말은 소비자 후속관리로 답하면 안 됨
- 여기서 파트너는 같이 사업할 사람이므로 사업 가이드/교육 연결/불안 해소로 답해야 함

### 7-4. 수정 필요한데 필요 없다고 오판하면 안 되는 항목

1. Script DB
   - 단순 카테고리 세분류가 아님
   - 발화자 역할에 따라 질문 예시/상황/판단 기준/말버전/키워드가 달라지므로 구조화가 필요

2. 케이스관리
   - 화면/분기 로직 구현은 나중이 맞음
   - 하지만 소비자/파트너/소개자B/스폰서A/C 역할에 따른 코칭 기준은 지금 Script/FAQ 후보로 정리해야 함

3. 전사본/Slack 원본 처리
   - 화자가 불명확하면 이름을 억지로 확정하지 말고 `화자 불명`, `동석자 추정`, `상담자 추정`으로 표시
   - 원본을 통째로 넣지 말고 발화자 역할과 사용 상황 기준으로 쪼개야 함

4. Slack+GENIA 운영모델
   - 이번 요청의 핵심은 서비스 권한 설계가 아니라 발화자 역할 구분임
   - Slack/지니아 채팅방 개발·권한 설계는 별도 기획 단계에서 다룰 것
