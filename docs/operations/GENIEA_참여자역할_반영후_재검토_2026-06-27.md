- 즉, 채팅방/전사본/스레드에서 누가 말하는지, 그 말이 어떤 맥락의 말인지 구분하는 기준이 필요하다.

## 2. 지금 구분해야 하는 것은 “사람 이름 전체”가 아니라 “발화자 역할”

모든 사람 이름을 다 DB/권한표로 구분할 필요는 없다.
아래처럼 발화 맥락에 영향을 주는 역할만 구분하면 된다.

| 발화자 역할 | 의미 | 왜 필요한가 |
|---|---|---|
| 김나연님 / 대표님 / 요청자 | Slack에서 지니아에게 지시·정정·기준을 주는 사람 | 운영 기준, DB 반영 기준, 말투 기준의 원천 |
| GENIEA/GD/봇 | 답변·정리·검토표 생성 주체 | 봇이 잘못 이해한 지점과 정정된 기준을 구분해야 함 |
| 매니저님 | 승인/확인 대상 | 실제 DB 반영, 대량 수정, 배포 전 확인 기준에만 중요 |
| 클라이언트/대표님이 언급한 현장 리더 | 예: 수정 리더님, 언주님 등 | 현장 자료 제공자, 체험사례/이미지/파트너 코칭 맥락에 영향 있을 때만 기록 |
| 파트너 | 같이 사업하는 사람 | 소비자 후속관리와 완전히 다른 답변이 필요함 |
| 소비자 | 제품을 먹거나 상담받는 사람 | 제품 상담, 후속관리, FAQ/Script 후보 추출에 필요 |
| 소개자/B | 소비자와 상담자 사이에서 반응을 전달하는 중간 역할 | 소비자→파트너 전환, ABC 코칭에서 중요 |
| 스폰서/A 또는 상위 리더 | 파트너/소비자 후원 구조에서 가이드하는 사람 | 라인십, B-C 분리/보호, 스폰서십 코칭에 필요 |
| C / 다운라인 / 상담 대상 | ABC 구조에서 실제 케어 받거나 사업을 이어갈 사람 | B와 C를 묶을지 분리할지 판단하는 맥락에 필요 |
| 동석자/화자 불명 | 전사본에서 누군지 불명확한 사람 | 억지로 이름 확정하지 말고 `화자 불명/동석자 추정`으로 둬야 함 |

핵심:
- 이름 자체가 중요한 게 아니라 `그 발화가 어떤 역할에서 나온 말인지`가 중요하다.
- 박정민/구유빈/김현지 같은 이름은 권한표 대상이 아니라, `파트너 상태별 코칭 사례`로만 기록하면 된다.
- 언주님은 사용자 권한 설계 대상이 아니라, `Image DB 자료 제공/분류 담당 맥락`으로만 기록하면 된다.
- 수정 리더님은 “김수정”이라는 개인 권한이 아니라, `파트너/리더 코칭 테스트 맥락의 발화자`로 이해하면 된다.

## 3. 누락 확인 방식도 바뀌어야 함

이제 누락 여부는 “사람 이름이 빠졌는가?”로 보면 안 된다.
아래 기준으로 봐야 한다.

### 3-1. 발화자 역할이 빠졌는지

확인할 것:
- 이 말은 누가 한 말인가?
- 실제 사용자의 지시인가, 봇의 해석인가?
- 소비자 발화인가, 파트너 발화인가, 스폰서/리더 발화인가?
- 전사본 속 화자가 확실한가, 추정인가?
- 역할이 바뀌면 답변도 바뀌는가?

예:
- “파트너가 생겼다”는 말은 소비자 후속관리로 답하면 안 됨
- 여기서 파트너는 `같이 사업할 사람`이므로 사업 가이드/교육 연결/불안 해소로 답해야 함

### 3-2. 역할이 바뀌면 필요한 DB가 달라지는지

예:
- 소비자 질문 → Product DB / FAQ DB / Link DB / 건강 상담 말버전
- 파트너 질문 → Script DB / 교육 연결 / 사람 떠올리기 / 상담 복기
- 스폰서 불만 질문 → 라인십 / ABC / 팀워크 / 감정 인정→감사 회복→장점 전환
- 이미지 자료 제공자 질문 → Image DB / 출처 / 동의 / 분류 / 사용범위

### 3-3. 이름을 기록해야 하는 경우

사람 이름은 아래 경우에만 기록한다.

1. 특정 사례의 상태가 다음 행동 판단에 중요할 때
   - 예: 박정민, 구유빈, 김현지 파트너 상태별 가이드

2. 자료 제공 역할이 중요할 때
   - 예: 언주님은 Image DB 업로드/현장 분류 방식 제공자

3. Slack 스레드 역할 구분에 필요할 때
   - 예: 김수정 스레드, 이언주 스레드, 파트너+지니아 미팅 스레드

4. 승인/책임 라인에 필요할 때
   - 예: 매니저님 `<@U9E0YD4JC>`은 실제 DB 반영 승인자

그 외에는 이름을 전부 나열할 필요 없다.

## 4. 수정 필요한데 필요없다고 오판한 것 — 정정 버전

### 오판 1. `참여자 역할`을 서비스 권한 설계로 확대함

정정:
- 지금 당장 필요한 것은 서비스 전체 권한 설계가 아니다.
- 필요한 것은 Slack/전사본/DB 후보 정리에서 `발화자의 역할 라벨`을 정확히 붙이는 것이다.

### 오판 2. 모든 사람 이름을 구분해야 하는 것처럼 정리함

정정:
- 모든 사람 이름을 구분할 필요 없다.
- 발화자가 DB 후보의 의미를 바꾸거나, 케이스 판단에 영향을 줄 때만 이름/상태를 기록한다.

### 오판 3. Slack+GENIA 운영모델을 역할/권한 설계 이슈로 과확장함

정정:
- Slack+GENIA 운영모델은 여전히 아이디어 누적 단계다.
- 다만 Slack 스레드별 `용도/발화자 역할`은 명확히 해야 한다.
- 개발/견적/권한 설계표를 지금 만들라는 뜻은 아니다.

### 오판 4. “운영자용 AI봇 권한”을 지금 산출물로 올림

정정:
- 운영자용 AI봇 권한 설계는 나중에 개발/운영모델 검토 시 볼 항목이다.
- 지금 요청의 핵심은 그게 아니라, 대화/전사본에서 발화자 역할을 정확히 읽고 DB 후보를 빠뜨리지 않는 것이다.

### 오판 5. 케이스관리/Script DB 기준은 여전히 중요함

이 부분은 유지한다.
단, 이유를 다시 쓴다.

- 케이스관리 화면/분기 기능 구현을 당장 하라는 뜻은 아님
- 하지만 발화자 역할이 소비자/파트너/스폰서/B/C 중 무엇인지에 따라 Script DB/FAQ 후보가 달라지므로, 정리 시 역할 라벨이 필요함

## 6. 최종 정정 요약

김나연님 질문에 대한 답:

- 소개자/B가 전달한 말인지
- 스폰서/A가 코칭하는 말인지
- 봇이 잘못 해석한 말인지
