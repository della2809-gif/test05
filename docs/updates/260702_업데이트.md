## 레퍼런스 파일 DB 한글 깨짐 대응

**왜 만들었나**
레퍼런스 파일 DB에서 일부 PDF 내용이 한글 인코딩 문제처럼 깨져 보이는 문제가 있었음. 확인 결과 DB가 한글을 잘못 저장한 것이 아니라, PDF 안의 텍스트/OCR 레이어를 자동 추출하는 과정에서 한글이 잘못 인식된 상태로 저장된 것이 원인이었음. 깨진 텍스트가 AI 검색(RAG)에 계속 사용되면 답변 근거 품질이 떨어질 수 있어, 원본 PDF는 보관하되 깨진 자동 추출 텍스트는 검색 대상에서 제외하도록 정리함.

**뭘 할 수 있나**
1. **깨진 PDF 추출 텍스트 검색 제외** — 기존 PDF 기반 레퍼런스 chunk를 삭제하고 PDF 레퍼런스 파일은 비활성 처리하여, AI가 깨진 텍스트를 근거로 답하지 않게 함
2. **원본 PDF 보관 유지** — PDF 파일 자체는 삭제하지 않고 관리자 레퍼런스 파일 목록에 남겨 둠. 추후 정제 TXT/OCR본으로 대체 업로드 가능
3. **정제 TXT/XLSX는 계속 사용** — 이미 깨끗하게 정리된 TXT/XLSX 레퍼런스는 활성 상태와 검색 chunk를 유지함
4. **향후 재발 방지** — 새 PDF 업로드 시 자동 추출 텍스트 품질이 낮으면 DB chunk 생성을 건너뛰고, 설명에도 “자동 추출 보류” 안내가 남도록 함

## 기술 변경 요약

| 파일 | 변경 내용 |
|------|----------|
| `src/lib/reference-quality.ts` | PDF 자동 추출 텍스트의 품질을 검사하는 함수 추가. 한글 OCR/폰트 매핑 오류 패턴, 이상 기호 비율, 대체 문자 등을 기준으로 RAG 저장 여부를 판단 |
| `src/lib/__tests__/reference-quality.test.ts` | 깨끗한 한글 텍스트는 통과하고, 깨진 PDF 추출 텍스트는 차단되는지 테스트 추가 |
| `src/app/api/admin/files/route.ts` | PDF 업로드 후 추출 텍스트가 품질 기준을 통과한 경우에만 description/chunk/embedding 저장. 품질이 낮으면 원본 PDF만 보관하고 자동 RAG 반영은 중단 |
| `scripts/cleanup_low_quality_reference_pdf_chunks.mjs` | 실서버에서 OCR/폰트 매핑 오류가 뚜렷한 PDF 레퍼런스 chunk를 백업 후 정리하는 스크립트 추가 |
| `scripts/deactivate_all_reference_pdf_chunks.mjs` | PDF 자동추출 chunk 전체를 백업 후 비활성/삭제하여, 정제 TXT/XLSX 중심으로 RAG를 운영하도록 전환하는 스크립트 추가 |

## 운영 참고

- PDF 원본은 아직 관리자 레퍼런스 목록에 남아 있으나 모두 비활성 상태임.
- 다시 검색에 사용하려면 PDF를 직접 활성화하지 말고, 먼저 정제된 TXT/OCR본을 업로드한 뒤 해당 TXT를 활성 레퍼런스로 사용하는 것이 안전함.
- 기존 백업 JSON에는 정리 전 description/chunk 원문이 보존되어 있으므로 필요 시 복구 가능함.
