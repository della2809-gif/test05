## 5월 29일 화상회의 스크립트 저장

**왜 만들었나**
5월 29일 회의에서 케이스 관리, 지플릿 노출 방식, 사용자가 따라갈 수 있는 업무 흐름에 대한 중요한 논의가 있었음. 이후 개발 맥락을 다시 확인할 수 있도록 회의 스크립트를 프로젝트 폴더 안에 정리해둘 필요가 있었음.

**뭘 할 수 있나**
1. **회의 내용 확인** — `03_화상회의` 폴더에서 5월 29일 회의 스크립트를 다시 볼 수 있음
2. **기획 근거 확인** — 케이스 관리 구조를 왜 바꿨는지 회의 원문 기준으로 확인 가능함

## 케이스 관리 — 단계형 업무 가이드 구조로 변경

**왜 만들었나**
기존 케이스 관리는 “수집 항목”과 “연결 지플릿” 중심이라, 사용자가 어떤 순서로 업무를 진행해야 하는지 한눈에 보기 어려웠음. 5월 29일 회의에서 클라이언트가 “초보자가 질문을 잘 못하더라도 순서대로 따라갈 수 있는 목차/가이드라인이 필요하다”고 요청했음.

**뭘 할 수 있나**
1. **업무 단계 추가** — 관리자 화면에서 케이스별로 1단계, 2단계, 3단계처럼 업무 흐름을 만들 수 있음
2. **단계 설명 입력** — 각 단계마다 사용자가 무엇을 해야 하는지 설명을 넣을 수 있음
3. **단계별 수집 항목 입력** — 각 단계에서 받아야 할 고객 정보나 체크 항목을 따로 적을 수 있음
4. **단계별 지플릿 연결** — 필요한 지플릿을 단계별로 연결할 수 있음

## 케이스 노출 순서 설정 추가

**왜 만들었나**
메인 화면에서 케이스나 지플릿이 의도한 순서대로 보이지 않으면, 클라이언트가 설계한 업무 흐름이 사용자에게 잘 전달되지 않음. 회의에서 “중요한 것부터 보이게 순서를 조절하고 싶다”는 요청이 있었음.

**뭘 할 수 있나**
1. **노출 순서 입력** — 관리자 케이스 관리 화면에서 숫자로 노출 순서를 정할 수 있음
2. **중요한 케이스 우선 표시** — 숫자가 작은 케이스가 사용자 화면에서 먼저 보이도록 설정 가능함

## 운영 DB 마이그레이션 적용

**왜 만들었나**
코드에서는 새 케이스 구조를 사용하도록 바뀌었지만, 운영 Supabase DB에는 아직 새 컬럼이 없어 저장 시 400 오류가 발생했음. 운영 서버에서도 같은 기능을 쓰려면 DB 구조를 함께 맞춰야 했음.

**뭘 할 수 있나**
1. **운영 DB 저장 가능** — 관리자 화면에서 새 케이스 구조를 실제 운영 DB에 저장할 수 있음
2. **업무 가이드 저장 가능** — `guide_steps`에 단계명, 설명, 수집 항목, 연결 지플릿이 저장됨
3. **노출 순서 저장 가능** — `sort_order` 값이 운영 DB에 저장됨

## 케이스 저장 테스트

**왜 만들었나**
화면만 바뀐 것이 아니라 실제 운영 DB에 저장되는지 확인해야 했음. 특히 이번 변경은 DB 컬럼 구조가 바뀌는 작업이라 저장 테스트가 필요했음.

**뭘 할 수 있나**
1. **새 케이스 저장 확인** — 테스트 케이스를 생성해 운영 DB에 정상 저장되는지 확인함
2. **업무 단계 저장 확인** — 단계명, 단계 설명, 수집 항목, 연결 지플릿이 저장되는지 확인함
3. **테스트 데이터 정리** — 검수 후 테스트용 `meeting_prep_test` 케이스는 삭제함

## 실제 케이스 데이터 정리

**왜 만들었나**
테스트 과정에서 임시 케이스가 남아 있으면 클라이언트가 실제 관리자 화면을 볼 때 혼란스러울 수 있음. 운영 DB에는 현재 사용할 케이스만 남겨야 했음.

**뭘 할 수 있나**
1. **미팅 사전 준비 사용** — `meeting_prep` 케이스가 실제 데이터로 유지됨
2. **신규 제품 상담 사용** — `new_product` 케이스가 실제 데이터로 유지됨
3. **임시 데이터 제거** — 테스트용 케이스는 삭제되어 화면이 깔끔하게 유지됨
