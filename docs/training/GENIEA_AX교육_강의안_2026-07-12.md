## 교육 목표 3가지

1. **일하는 방식의 변화** — AX가 되어 팀원들이 실제 업무에 AI를 다르게 쓰기 시작한다.
2. **지니아 가치의 이해** — 지니아라는 프로덕트가 어떤 방식으로, 왜 이렇게 만들어졌는지 이해한다.
3. **인수인계를 위한 개발 기초** — 프로덕트를 넘겨받아 운영하기 위해 "AI에게 질문하고 결과를 검수할 수 있는 수준"의 개발 어휘를 갖춘다.

> 성공 기준: 교육이 끝났을 때 참가자가 "내가 AI를 어떻게 더 잘 쓸 수 있는지 알겠다"는 감각을 갖고 나가면 충분하다.

## 0. 오프닝 — 지금 AI 어떻게 쓰세요? (15분)

**목표**: 참가자의 현재 위치 파악 + 마지막 실습의 재료 수집

**진행**
- 한 명씩 돌아가며: "어떤 업무에, 어떤 AI를, 어떻게 쓰고 계세요?"
- 답변을 화이트보드에 전부 기록한다. **이 목록은 지우지 않는다** — 10번 실습에서 다시 쓴다.
- 안 쓰는 사람이 있어도 괜찮다는 분위기: "오늘 끝나면 뭐부터 시킬지 정해서 나가시면 됩니다."

**팁**
- 여기서 나온 답변의 수준(단순 검색 vs 문서 작성 vs 자동화)이 이후 강의의 눈높이 조절 기준이 된다.

## 1. AI의 변화 — 채팅에서 "딸깍"까지 (10분)

**목표**: 지금이 어떤 변곡점인지 체감시키기

**내용**
- AI의 역사 압축: 검색의 시대 → ChatGPT의 등장 (각자 "GPT 처음 쓰고 wow 했던 순간" 한 마디씩)
- 변화의 단계:
  - **채팅** — 물어보면 답해준다 (내가 일하고, AI는 조수)
  - **에이전트** — 시키면 알아서 해온다 (AI가 일하고, 나는 검수자)
  - **그 다음은?** — 시키지 않아도 알아서 발견하고 제안한다 (질문 던지고 참가자 의견 듣기)
- 이 모든 변화를 관통하는 욕망: **"딸깍만 하고 싶다. 제발 알아서 해줘!"**
  - 우스갯소리 같지만 이게 수조 원이 움직이는 방향이라는 것.

**연결 멘트**: "그런데 '알아서 해줘'가 실제로 되려면 뭐가 필요할까요?" → 2번으로

## 2. 컨텍스트와 위임의 기술 — 일 잘하는 팀원 만들기 (30분)

**목표**: AI 활용의 근본 원리인 **컨텍스트** 개념을 먼저 깔고, 그 위에서 "위임 잘하기 = 컨텍스트 잘 주기"임을 깨닫게 하기

### 2-1. 컨텍스트 — AI는 아는 만큼만 일한다 (10분)

- **정의**: 컨텍스트 = AI가 답을 만들기 직전에 손에 쥐고 있는 모든 정보. 내 지시문 + 지금까지의 대화 + 첨부한 파일과 문서, 전부.
- **공식 하나로 요약**:
  > **AI의 실력 = 모델 성능 × 컨텍스트 품질**
  >
  > 모델은 누구나 같은 걸 쓴다. 그런데 왜 어떤 사람은 AI로 대단한 걸 만들고 어떤 사람은 뻔한 답만 받을까? — 차이는 전부 컨텍스트에서 난다.
- **비유**: 엄청나게 똑똑하지만 **우리 회사를 전혀 모르는 알바생**. 세상 지식은 다 아는데 지니아의 고객, 상품, 내부 규칙은 하나도 모른다. 첫날 "알아서 해줘"가 실패하는 이유.
- **중요한 사실**: AI는 지니아의 내부 사정을 모른다 — 우리가 컨텍스트로 **넣어준 만큼만** 일한다. (넣어주지 않은 걸 물으면 어떻게 되는지는 7번에서 — 환각의 복선)
- **한계도 있다**: 컨텍스트는 무한하지 않다. 비유하면 **책상 크기** — 아무리 좋은 서류도 책상에 다 올릴 수는 없다. 그래서 "뭘 올릴지 고르는 것"이 기술이다. (5번 '기록을 잘 쌓는 법'의 복선)
- **"새 채팅을 열면 왜 다 까먹어요?"** — 새 세션 = 책상을 싹 치우고 새로 시작하는 것. 그래서 매번 다시 설명하기 싫으면 어딘가에 적어둬야 한다. (3번 메모리·지침의 복선)
- **연결 멘트**: "그러면 그 책상 위에 **뭘** 올려줘야 일을 잘할까요?" → 2-2로

### 2-2. 위임의 기술 (20분)

**진행 (참여형)**
- 질문: "내일 신규 팀원이 입사합니다. '이거 해줘~' 한 마디로 일을 잘하게 만들려면, 그 사람에게 뭘 줘야 할까요?"
- 답변을 화이트보드에 받아 적는다. 아래 8가지가 나오도록 유도:

| 항목 | 설명 |
|------|------|
| **일의 목적/배경** | 왜 이 일을 하는지, 결과물이 어디에 쓰이는지 |
| **일의 원칙** | 우리 팀이 뭘 중요하게 여기는지 (속도보다 정확성? 고객 톤앤매너?) |
| **의사결정 기준** | 애매한 상황에서 뭘 우선할지 (A/B 갈리면 뭘 고르는지) |
| **실행 순서** | 어떤 절차로 하는지, 어디서 자료를 찾는지 |
| **좋은 결과물의 예시** | "지난번 이거처럼 해줘" 할 수 있는 샘플 |
| **금지사항** | 절대 하면 안 되는 것 (고객에게 임의 약속 금지 등) |
| **완료 기준** | 언제 "끝났다"고 판단하는지, 누가 검수하는지 |
| **보고/피드백 방식** | 중간에 막히면 언제 어떻게 물어보는지 |

**결정적 한 방**
> "이 목록이 그대로 AI에게 주는 **컨텍스트**의 구성요소입니다. 신규 팀원 온보딩 문서 = AI 시스템 프롬프트. 여러분은 이미 위임을 할 줄 압니다. 대상이 사람에서 AI로 바뀌었을 뿐입니다."

**팁**: 이 표는 인쇄해서 나눠주면 10번 실습 시트가 된다.

## 3. 모두가 같은 생각을 하게 만드는 도구들 (20분)

**목표**: 개인의 프롬프트를 넘어 "팀 전체가 공유하는 AI 세팅"이 있다는 것을 알려주기

**내용**
- 문제 제기: "각자 프롬프트를 잘 쓰면 끝일까요? 팀원 5명이 AI한테 5가지 다른 지시를 하면?"
- 사람 세계의 장치와 AI 세계의 장치를 1:1 매핑:

| 사람 세계 | AI 세계 |
|-----------|---------|
| 온보딩 문서, 팀 그라운드룰 | **시스템 프롬프트 / 프로젝트 지침** (CLAUDE.md 등) |
| 직무 정의, 담당 역할 | **페르소나 / 서브에이전트** |
| 업무 매뉴얼, SOP | **스킬** (반복 업무를 절차로 패키징) |
| "이럴 땐 꼭 이렇게" 자동 규칙 | **훅** (특정 시점에 자동 실행) |
| 팀원의 경험과 기억 | **메모리** (세션이 바뀌어도 유지) |
| 회의 하나의 맥락 | **세션 / 컨텍스트** |
| 사내 위키, 문서고 | **지식 저장소 / RAG** |
| 사내 시스템 접근 권한 | **MCP / 도구 연결** (슬랙, 드라이브, DB…) |

**진행 팁**
- 왼쪽 열(사람 세계)을 먼저 보여주고 오른쪽을 하나씩 공개하는 방식이 효과적.
- 각 항목마다 실제 예시 한 줄씩: "훅 예시 — 문서를 저장할 때마다 자동으로 맞춤법 검사를 돌린다."
- 용어를 외우게 하는 게 목적이 아니라 "이런 게 존재한다"는 지도를 그려주는 것이 목적.

## 5. 클로드 코드 & 클로드 코워크 — 폴더에서 일하는 AI + 기록을 잘 쌓는 법 + 보안 (25분)

**목표**: 채팅형 AI와 에이전트형 AI의 결정적 차이인 **"디렉토리(폴더) 기반"** 개념을 이해시키고, 그 폴더에 기록을 **md 문서로, 망하지 않게** 쌓는 방법까지 가르친 뒤, 실습(10번)에서 쓸 도구를 소개하기

**내용**

- **채팅형 AI의 한계부터**: 대화가 끝나면 맥락이 사라진다. 결과물은 복사-붙여넣기로 옮겨야 한다. 자료를 매번 다시 설명해야 한다. → "매번 처음 만난 사람에게 전화로 일 시키는 것"
- **디렉토리 기반이란**: AI에게 채팅창이 아니라 **작업 폴더를 통째로 준다**
  - 폴더 안의 파일이 곧 맥락이다 — 지침 문서, 참고 자료, 이전 산출물을 AI가 직접 읽는다
  - AI가 일한 결과도 **파일로 폴더에 남는다** — 다음 세션의 AI가, 다른 팀원이, 그대로 이어받는다
  - 비유: "전화 통화로 일 시키기" vs **"신규 팀원에게 책상, 캐비닛, 업무 매뉴얼을 통째로 주기"**
  - 4번과 연결: **온톨로지는 바로 이 폴더에 쌓인다.** 지침 파일, 결정 기록, 산출물이 폴더에 누적되는 것 = 온톨로지가 만들어지는 실제 과정
- **두 가지 도구**:

| | 클로드 코드 (Claude Code) | 클로드 코워크 (Claude Cowork) |
|---|---|---|
| 누구용 | 개발자 | 비개발자 (일반 업무) |
| 작업 대상 | 코드 폴더 | 문서·엑셀·자료 폴더 |
| 하는 일 | 코드를 읽고, 고치고, 기능을 만든다 | 문서를 읽고, 정리하고, 산출물을 만든다 |
| 지니아와의 관계 | **지니아가 이걸로 만들어졌고, 유지보수된다** | 여러분이 일상 업무에 바로 쓸 도구 |

- **인수인계 연결 (중요)**: "지니아 인수인계란 결국 **이 폴더를 통째로 넘겨받는 것**입니다. 코드, 문서, 그동안의 결정 기록이 전부 폴더 안에 있고, 클로드 코드가 그걸 읽으면서 여러분의 질문에 답해줄 겁니다."

### 5-2. 기록을 "잘" 쌓는 법 — md 문서의 기술

**훅 (대표님 이야기로 시작)**
> "대표님이 회의를 전부 녹음하고, 기록하고, 쌓아두시는 걸 보셨을 겁니다. 사실 이게 온톨로지의 절반입니다. 나머지 절반은 그 기록을 **AI가 읽을 수 있는 형태로, 잘** 쌓는 것입니다."

**① 왜 md(마크다운) 문서인가**
- md = 제목, 목록 정도의 구조만 있는 순수 텍스트 파일. **AI가 가장 잘 읽고, 직접 수정도 할 수 있는 형식**
- 워드/한글/PDF는 사람용, md는 사람과 AI 공용
- 흐름: 녹음 → 텍스트 변환 → **AI에게 "md로 정리해줘"** → 폴더에 저장 (이 정리 자체도 AI에게 딸깍 시키는 일)

**② 그런데 무한정 쌓으면 망한다**
- AI가 한 번에 읽을 수 있는 양(컨텍스트)에는 **한계가 있다** — 2-1에서 배운 "책상 크기"의 제약
- 전부 다 읽게 하면: 느려지고, 비싸지고, 오래된 정보와 새 정보가 섞여 **중요한 걸 놓친다**
- 비유: "창고에 다 던져넣기" vs "정리된 서재" — 서재에는 목차가 있고, 안 보는 책은 서고로 간다

**③ 잘 쌓는 5가지 원칙**

| 원칙 | 방법 |
|------|------|
| **원본과 정리본 분리** | 녹음 전문은 아카이브 폴더에. AI에게 주는 건 "결정과 이유"만 추린 정리본 |
| **한 문서 = 한 주제** | 만능 대형 파일 하나 대신, 작은 파일 여러 개 |
| **목차(인덱스) 파일** | INDEX.md에 문서당 한 줄 요약 — AI가 목차만 보고 필요한 문서만 찾아 들어간다 |
| **날짜와 상태 명시** | 파일명에 날짜(`2026-07-12_결정요약.md`), 뒤집힌 결정은 취소선 처리 후 아카이브로 |
| **새로 쓰기 전에 검색** | 같은 주제 문서가 있으면 새로 만들지 말고 업데이트 — 중복·모순은 AI를 헷갈리게 한다 |

- **핵심 기준**: 무엇을 남기나? "무엇을 했다"가 아니라 **"왜 그렇게 결정했다"**. 결정과 이유가 쌓여야 AI가 다음 판단을 대신할 수 있다.
- **실물 공개**: 지니아 폴더의 `docs/`(날짜별 업데이트 기록)와 결정 기록 구조를 화면으로 보여준다 — "이미 이렇게 쌓이고 있습니다."

### 5-3. 보안 — AI에 넣으면 안 되는 것 (3분, 필수)

**프레이밍**: "AX가 되는 순간 제일 먼저 나는 사고가 이겁니다. 딱 세 가지만 기억하세요."

1. **회사 계정으로만 쓴다** — 개인 무료 계정에 회사 자료를 넣지 않는다 (무료 계정은 입력 내용이 AI 학습에 쓰일 수 있음)
2. **고객 정보는 가리고 넣는다** — 이름·연락처·주민번호·건강정보가 담긴 파일을 통째로 붙여넣지 않는다. 필요하면 "홍길동 → 고객A"로 치환하고 사용
3. **애매하면 넣기 전에 묻는다** — "이거 넣어도 되나?" 싶으면 이미 답은 '먼저 확인'

- 지니아 도메인 특성상 고객의 건강·보험 정보가 오간다는 점을 콕 집어서 강조할 것.
- "이 세 가지 때문에 AI를 겁내지는 마세요. 원칙만 지키면 안전합니다"로 마무리 (겁주기가 목적이 아님).

**진행 팁**
- 가능하면 1~2분 라이브: 코워크에 예제 폴더를 주고 "이 폴더 안 문서들 요약해줘" 한 마디 → 폴더를 읽고 결과 파일을 만들어내는 장면. (10번 실습의 예고편)

## 6. 스킬 생태계와 역전 (10분)

**목표**: 개인/회사를 넘어 시장 차원의 변화 → "역전"의 충격 전달

**내용**
- **스킬이 시장에 나타나기 시작했다**: 잘 만든 업무 매뉴얼(스킬)이 공유되고 거래된다. 남이 만든 "일 잘하는 법"을 다운로드해서 내 AI에게 장착하는 시대.
- 그 결과 — **역전**:
  - "개발자가 아닌데 개발을 한다?"
  - "디자이너가 아닌데 디자인을 한다?"
  - 실행 능력이 더 이상 진입장벽이 아니게 됐다.
  - 5번에서 본 클로드 코드가 바로 그 도구 — 폴더와 지시만 있으면 실행은 AI가 한다.

## 7. 그렇다면 중요한 것은? (15분)

**목표**: 역전의 시대에 살아남는 능력이 무엇인지 — 핵심 메시지 전달

**먼저 찬물 한 바가지 — 환각(할루시네이션) (3분)**
- "AI가 이렇게 대단한데, 왜 사람이 필요할까요? AI에게는 치명적인 습관이 하나 있습니다."
- **AI는 모르면 '모른다'고 하지 않고, 그럴듯하게 지어낸다.** 그것도 아주 자신 있는 말투로. (2-1 회수: 컨텍스트에 없는 것은 모르는 것인데, 모른다고 말하지 않는다)
- 실례로 보여주기 (미리 준비): 존재하지 않는 보험 약관 조항이나 특약을 물으면 AI가 그럴듯한 조항 번호와 내용을 지어내는 장면. 라이브가 부담되면 캡처 화면으로.
- 운영 관점 경고: "인수인계 후 '**AI가 그렇다고 했어요**'는 절대 면책이 되지 않습니다. 고객에게 나가는 정보는 반드시 사람이 원문을 확인합니다."
- 연결: "그래서 AI 시대에 사람의 역할이 사라지는 게 아니라, **바뀌는** 겁니다." → 핵심 메시지로

**핵심 메시지**
> **실행이 싸지면, 판단과 맥락이 비싸진다.**

실행(코딩, 디자인, 문서 작성)은 AI가 한다. 사람에게 남는 것 세 가지:

1. **무엇을 만들지 아는 것** — 문제 정의, 도메인 지식
2. **좋고 나쁨을 알아보는 눈** — 결과물 검수 능력, 취향 (방금 본 환각을 걸러내는 눈이 바로 이것)
3. **맥락을 언어화하는 능력** — 머릿속 암묵지를 AI가 읽을 수 있는 문서로 꺼내는 것

**슬로건**
> "개발자를 이기는 게 아니라, **자기 도메인을 가장 잘 언어화한 사람이 이긴다.**"

**연결 멘트**: "이 말이 진짜인지, 증거를 보여드리겠습니다." → 8번으로 (이 섹션은 8번의 복선)

## 8. 지니아를 구축한다는 것 (15분)

**목표**: 목표 2 (지니아 가치 이해) 달성 + 감정적 정점

**내용**
- 대표님 이야기: **이 스트림(개발)을 모르는 사람이 이걸 구축했다.**
- 어떻게 가능했나 — 7번의 세 가지가 그대로 있었기 때문:
  1. 무엇이 필요한지 알았다 (현장의 문제, 도메인 지식)
  2. 결과물이 맞는지 틀린지 알아봤다 (검수의 눈)
  3. 그것을 집요하게 언어로 꺼냈다 (요구사항, 피드백, 수정 요청)
- **기록 습관 회수 (5-2 연결)**: 대표님이 전부 녹음하고 기록하고 쌓아온 것 — 그게 바로 3번(언어화)의 실천이었고, 그 기록이 md 문서가 되어 지니아 폴더에 온톨로지로 쌓였다. "특별한 재능이 아니라 **기록하는 습관**이 이걸 만들었습니다."
- 지니아 프로덕트 실물 시연: 상담 화면, 관리자 페이지 등을 보여주며 "이게 그 결과물입니다."
- **참가자에게 던지는 메시지**: "대표님이 특별해서가 아니라, 방법이 존재해서 가능했던 겁니다. 그 방법을 오늘 여러분이 배웠고, 이제 이 프로덕트를 여러분이 이어받습니다."

## 9. 인수인계를 위한 개발 상식 (20분)

**목표**: 목표 3 달성 — 직접 개발하는 수준이 아니라 **"AI에게 질문하고 결과를 검수할 수 있는 어휘"** 갖추기

**프레이밍 (중요)**
> "여러분이 코드를 짤 일은 없습니다. 하지만 AI에게 '이거 고쳐줘'라고 시키려면, 뭐가 어디에 있는지 말할 수 있어야 합니다. 오늘은 그 지도를 드립니다."

**내용 — 전부 지니아 실물로 설명한다** (추상 개념 금지)

| 개념 | 지니아 실물 예시 |
|------|-----------------|
| **프론트엔드** | 여러분이 매일 보는 화면 — 상담 화면, 관리자 페이지의 버튼과 목록 |
| **백엔드** | 화면 뒤에서 답변을 만들어주는 부분 — 질문을 받아 답을 조립하는 로직 |
| **DB (데이터베이스)** | 스크립트, FAQ, 상담 기록이 저장되는 창고 — 관리자 페이지에서 수정하면 여기가 바뀐다 |
| **API** | 프론트와 백엔드가 대화하는 창구 — "이 질문에 답해줘"라는 주문서가 오가는 길 |
| **외부 AI 연동** | 지니아의 답변 생성은 OpenAI 같은 외부 AI를 호출해서 이뤄진다 — 그래서 사용량에 따른 비용이 발생한다 |
| **RAG / 지식 검색** | AI가 답하기 전에 우리 스크립트·FAQ를 먼저 찾아보고 답하는 구조 — 그래서 스크립트 DB 관리가 답변 품질을 좌우한다 |
| **배포** | 수정한 코드가 실제 서비스에 반영되는 과정 — "고쳤는데 왜 안 바뀌어요?"의 답 |
| **깃 (버전 관리)** | 모든 변경의 기록 — 언제 무엇이 왜 바뀌었는지 되돌아볼 수 있는 타임머신 |
| **레포지토리 (작업 폴더)** | 지니아의 코드·문서·결정 기록이 전부 담긴 폴더 — 5번에서 배운 "디렉토리". **인수인계의 실체가 바로 이 폴더다** |

**운영 관점 시나리오 연습 (구두로)**
- "답변이 이상하게 나온다" → 어디부터 볼까? (스크립트 DB? 프롬프트? AI 자체?)
- "관리자 페이지에서 저장했는데 반영이 안 된다" → 어떤 질문을 해야 할까?
- 포인트: 정답을 맞히는 게 아니라 **"어디를 의심할지 어휘로 말할 수 있게 되는 것.**"

**팁**: 이 표를 유인물로 만들어 인수인계 기간 내내 책상에 두게 하면 좋다.

## 10. 라이브 실습 — 딸깍 해보기 (20분)

**목표**: 목표 1 달성 — "나도 할 수 있다"를 강의장 안에서 실제로 경험시키기

**형식**: 노트북 1대 + 화면 미러링. 대표 1명이 조작하고 전원이 함께 설계한다. 도구는 **클로드 코워크** (준비된 실습 폴더 사용 — 5번에서 배운 디렉토리 기반을 실제로 체험).

**진행**
1. **(3분) 과제 선정** — 오프닝 때 화이트보드에 적어둔 업무 목록에서, 다 같이 투표로 하나 고른다. "AI에게 딸깍 시키고 싶은 것" (예: 반복 문서 정리, 고객 안내문 초안, 간단한 계산 페이지)
2. **(7분) 지시문 공동 작성** — 2번에서 배운 위임 프레임 8항목을 다 같이 채운다. 강사가 화이트보드에 받아 적고, 대표가 타이핑. **"프레임이 있는 지시"와 "한 줄 지시"의 차이를 몸으로 느끼게 하는 단계.**
3. **(8분) 실행 & 관전** — 실제로 돌린다. 결과가 나오는 과정을 다 같이 본다. 첫 결과가 아쉬우면 오히려 좋다 — "완료 기준을 더 명확히 줘볼까요?" 하고 지시문을 고쳐 재실행 (검수→재위임 루프 시연).
4. **(2분) 정리** — "방금 하신 게 위임→실행→검수→재위임입니다. 이게 앞으로 여러분이 일하는 방식입니다."

**백업 플랜**: 투표로 고른 과제가 라이브로 돌리기 애매하면, 미리 준비한 안전한 과제(예: "지니아 FAQ 초안 10개 만들기")로 전환한다. **미리 한 번 리허설해 둘 것.**
