# 의사결정: 자동견적 OCR 규칙 데이터화 — 건강체크표 사진 → 판정 → 3단 견적 카드

## 배경
- 클라이언트가 자동견적의 핵심으로 지목한 "건강체크 그래프 사진(OCR) → 자동 분석 → 패키지 선택 → 견적 카드" 플로우가 미반영 상태였음 (7/7 로직테스트 스레드).
- 클라이언트가 최종 기준안을 확정해 전달함: `00_requirements/0707_자동견적로직_매니저님전달최종.txt` (등급별 구성표, 기본/메가/베이직/리셋 구성, 건강축별 추천 제품표, 추가 로직, 분석 문장 톤, 견적 카드 항목).
- 기존 이미지 경로는 `health-analyzer`+`quotation-engine`(__QUOTATION__) 조합으로, 확정 기준안의 등급별 3단(프리미엄/스탠다드/베이직) 구성표·축별 추가 제품 로직과 다름.

## 선택지
- (a) LLM 프롬프트에 기준안을 넣고 LLM이 구성·수량·금액을 산출 / (b) 기준안 전체를 규칙 데이터 모듈로 옮기고 조합·수량·금액은 전부 결정적 코드로 계산 (LLM은 OCR 점수 인식까지만)

## 결정
- **(b) 규칙 데이터화 채택.**
  - `src/lib/health-quote-rules.ts` — 기준안 원문을 그대로 데이터로 옮긴 단일 소스: 축 이름(A~J), 등급별 3단 구성표(스펙 3장), 기본영양제/메가영양제/베이직/리셋1주/리셋2주 구성(4·5장), 뉴트리밀 선택지·맛(11장), 건강축별 추천 제품표(6장), 추가 수량 기준(7장: 양호0/보통0/경계1/불량2), 오토십 10%·할인 제외 품목(쉐이커/지퍼백/셀라비브).
  - `src/lib/health-quote-engine.ts` — 판정·조합 엔진: OCR 색구간(양호/보통/경계/불량) 우선, 점수 구간 폴백 → 전체 등급 = 최악 축 → 등급별 블록 병합(리셋+영양제 수량 합산) → 축별 추가 제품(불량>경계 순, 공통 제품 우선, 기본 구성·리셋 뉴트리밀 포함 제품은 중복 추가 금지, 축 간 중복은 축명 병기로 정리) → 3단 견적. 제품 단가는 package-quote-engine의 `resolveProduct`(별칭 해석) 재사용, 미해석은 "단가 미확인" 정직 표기.
  - 건강 분석 문장(스펙 9~10장)도 결정적 템플릿: 진단/치료 표현 금지, 양호·보통 리셋=유지(안티에이징) / 경계·불량 리셋=회복 목적 구분.
  - 출력: `__HEALTHQUOTE__` 마커 → `src/components/chat/health-quote-card.tsx` (① 건강 분석 카드 ② 3단 견적 카드, 뉴트리밀/맛/추가선택, 예상 첫 수당·적립금은 "예상" 표기, 카카오 주문 메시지 복사, consultations 저장).
  - OCR이 점수·색구간을 하나도 못 읽으면 견적을 만들지 않고 A~J 점수를 텍스트로 되묻는다 (지어내기 금지).
- 기존 __QUOTATION__ 경로는 새 경로가 대체하되, 렌더러·엔진은 남겨 과거 메시지 표시 회귀를 막음.

## 이유
- 구성·수량·금액은 지어내면 안 되는 값 — LLM 산출은 매번 결과가 달라지는 B-2와 같은 문제를 재생산함. 기준안이 표 형태로 확정돼 있어 데이터+결정적 코드로 1:1 대응 가능하고, 스펙 대조·단위 테스트로 검증 가능.
- 규칙을 한 파일에 모아 클라이언트 기준안 개정 시 rules 파일만 수정하면 됨.

## 검증 기준
- 건강체크표 사진 업로드 → ① 건강 분석 카드(축별 판정 칩 + 상담용 문장) ② 프리미엄/스탠다드/베이직 견적 카드가 순서대로 출력된다
- 불량 축 제품 2개·경계 축 제품 1개가 추가되고, 기본 구성(리셋 포함 시 뉴트리밀 포함)에 이미 있는 제품은 추가 제품 목록에 없다
- 두 축의 공통 제품은 한 줄로만 추가되고 축명이 병기된다
- 단가 미확인 구성품은 "단가 미확인"으로 표시되고 합계에서 제외된다고 안내된다
- 수당·적립금 문구에 "예상" 표현이 항상 포함된다
- 점수를 못 읽은 사진은 견적 없이 A~J 점수를 텍스트로 되묻는다
