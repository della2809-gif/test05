# 의사결정: CR-037 자유채팅 제품DB — 제품 로더 게이트를 1순위/2순위로 분리

작성일: 2026-06-27

## 배경
- CR-037: 자유 채팅(general/catch-all)에서 "헬스팩 알려줘"처럼 물으면 제품DB 기반 상세정보·가격·유사IQ 링크가 아니라 일반 종합영양제 설명처럼 나옴.
- 원인: `src/lib/openai.ts` 제품 로더의 게이트 `if (!isCatchAll || hasKeyword(userLastMessage, PRODUCT_KW))`가 자유채팅에서 PRODUCT_KW(제품/영양제/비타민/오메가/유사나/복용/성분/효능/가격/구매/건강기능/추천)가 없으면 **1순위 정밀검색까지 통째로 차단**.
- `extractSearchTerms("헬스팩 알려줘")`는 `["헬스팩"]`을 정상 추출하고, 제품 검색은 이미 `aliases`(별칭) 필드까지 ilike 검색하므로, 게이트만 풀리면 별칭으로 정확히 매칭됨.
- 이 게이트는 CR-007(`00_decisions/2026-06-26_search_cr004-007-008-...md`)에서 의도적으로 넣은 정책. 목적은 *자유채팅에 무관한 제품을 매 메시지 통째로 덤프하는 것 방지*(토큰·정확도). → 함부로 제거하지 않고 의도를 보존하는 방식 선택.

## 선택지
- A. 게이트를 1순위/2순위로 분리 (1순위 정밀검색은 항상 실행, 2순위 폴백 덤프만 게이트 유지)
- B. 제품 상세질문을 제품정보검색 지플릿으로 유도 (모드 전환 필요, UX 손해)
- C. 프롬프트로만 단정 금지 (DB 미주입이라 가격·링크 못 줌, 단독 불충분)

## 결정
- **A 채택 (정밀구현).** 김나연님 컨펌(2026-06-27).
- 제품 로더(`src/lib/openai.ts`)를 다음과 같이 분리:
  - **1순위 정밀검색**(searchTerms로 name/keywords/symptoms/category/tags/aliases ilike): `isCatchAll` 무관하게 **항상 실행**. 실제 매칭될 때만 주입하고 early return.
  - **2순위 최근제품 폴백 덤프**(관련성 없는 15개): 기존 게이트 `!isCatchAll || hasKeyword(PRODUCT_KW)` **유지** → CR-007 의도 보존.

## 이유
- 1순위는 실제 DB 매칭이 있을 때만 주입하므로, 자유채팅 잡담("오늘 뭐하지")엔 아무것도 안 들어가 토큰·정확도 회귀 없음. "헬스팩 알려줘"는 별칭 매칭으로 제품·가격·링크 정상 출력.
- 2순위 덤프는 그대로 막아 CR-007이 우려한 "무관한 제품 통째 덤프"를 방지.
- 제품 상세링크는 제품 row의 `usana_iq_url`에 포함되어 주입되므로, 별도 links DB 게이트(LINK_KW)는 건드릴 필요 없음.
- B/C는 "GENIEA가 DB를 알고 답하는 느낌"(클라이언트 의도)·완성도 측면에서 A보다 약함.

## 영향 범위
- `src/lib/openai.ts` — `if (dbSources.includes("products"))` 제품 로더 블록만 수정. 다른 로더(templates/blocks/links/images/stories 등) 미변경.
- 전용 지플릿(제품정보검색 등, `isCatchAll=false`) 동작은 기존과 동일(1순위 매칭 시 주입, 미매칭 시 2순위 덤프).

## 검증 기준
- 자유채팅에서 "헬스팩 알려줘" → 제품명·가격·유사IQ 상세링크가 출력된다.
- 자유채팅 잡담(제품명/별칭 미포함) → 제품 정보가 주입되지 않는다(무관한 제품 덤프 없음).
- 전용 제품정보검색 지플릿의 기존 출력은 회귀 없이 유지된다.
- 타입체크 통과, ESLint 신규 이슈 0 (베이스라인 동일).

## 되돌리기
- 쉬움. 단일 블록 변경이라 항목 revert 가능. 1순위를 다시 게이트 안으로 넣으면 원복.
