# 의사결정: 관리자 목록 서버사이드 검색+페이지네이션 (1,000행 한도 사고 대응)

## 배경
- 클라이언트가 "직접 입력한 DB가 유실됐다"고 보고 (7/5 저녁, #geniea).
- 조사 결과 **데이터 유실 아님**: 6/29 백업 678행 전부 라이브 존재, 4/22 입력분까지 생존 확인. 6/30 반영 스크립트에 DELETE 없음.
- 실제 원인: 관리자 Script DB/FAQ 화면이 limit 없이 전체 select → Supabase(PostgREST)가 **최대 1,000행만 반환**. 최근 수정순 정렬이라 6/30 대량 반영분이 1,000자리를 독차지 → 예전 수동 입력분 4,803행이 화면·검색(클라이언트-사이드 필터)에서 보이지 않았음.

## 결정
- admin_blocks(5,833행)·faqs(1,183행) 화면을 **서버사이드 검색(or ilike: 제목/카테고리/내용)·카테고리 필터·PAGE_SIZE 100 페이지네이션("더 보기") + 총 건수 표시**로 전환.
- 카테고리 바 목록은 화면에 로드된 일부가 아니라 전체 행에서 1,000행씩 페이지네이션해 수집.
- 검색어는 PostgREST or() 구분자 충돌 문자(`,()%`) 제거 후 사용.

## 이유
- 전량 로드는 1,000행 한도 때문에 불가능하고, 5,800행 전체 본문을 브라우저로 내리는 것도 과함. 서버 필터+페이지가 정확성과 성능 모두 해결.

## 영향 범위
- `src/app/admin/blocks/page.tsx`, `src/app/admin/faqs/page.tsx`
- 다른 관리자 테이블은 현재 1,000행 미만(stories 469, images 265, files 182 등) — 초과 시 같은 패턴 적용 필요.

## 검증 기준
- Script DB 화면에 "총 5,8xx건 중 100건 표시"가 뜨고, 검색으로 4월 입력 자료(예: "네트워크 마케팅(유사나)이란")가 찾아진다.
- FAQ 화면 동일. "더 보기"로 이어서 로드된다.
